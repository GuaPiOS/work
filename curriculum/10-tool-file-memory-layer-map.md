# Tool / File / Memory 三层分工表

## 一、总原则

这三层不是替代关系，而是分工关系：

- Tool 负责行动
- File 负责外置与发现
- Memory 负责跨时间知识组织与正确取回

如果把三层混成一层，agent 往往会出现两个问题：

1. 该外置的内容一直留在上下文里，导致膨胀
2. 该被结构化检索的知识只被当成普通文件，导致取回不准

## 二、三层定义

| 层 | 核心问题 | 主要职责 | 典型载体 | 风险 |
| --- | --- | --- | --- | --- |
| Tool | 我现在能做什么动作？ | 调用外部能力、改状态、拿结果 | MCP tools、CLI、API wrappers | 工具太多、描述模糊、恢复路径差 |
| File | 我怎么把大量信息外置并按需找回？ | 持久化中间结果、计划、日志、模块内容 | Markdown、JSONL、YAML、txt | 堆积无结构、不可发现、命名混乱 |
| Memory | 我如何跨时间稳定记住并正确取回知识？ | 维护偏好、实体、关系、时效、长期事实 | 文件系统、KV、vector store、graph DB | 取回不准、事实过期、实体混乱 |

## 三、什么时候该用 Tool

适合放在 Tool 层的内容：

- 需要发起动作
- 需要调用外部服务
- 需要执行命令、搜索、读写接口
- 结果本身不是最终记忆，而是当前行动的反馈

典型例子：

- 搜索 X 数据
- 运行测试
- 获取用户记录
- 生成评分结果

不要把 Tool 当成：

- 长期知识仓库
- 巨型上下文容器
- 代替文件系统的存储层

## 四、什么时候该用 File

适合放在 File 层的内容：

- 大体量工具输出
- 当前计划与 checklist
- 分模块技能内容
- 需要按需搜索和局部读取的文档
- 子代理共享结果

典型例子：

- `scratch/tool_outputs/*.txt`
- `plans/current_plan.yaml`
- `agents/research/findings.md`
- `content/posts.jsonl`

不要把 File 误解成：

- 自动等于 memory
- 只是临时过渡层
- 只能存静态文档

## 五、什么时候该用 Memory

适合放在 Memory 层的内容：

- 用户偏好
- 跨会话事实
- 实体身份信息
- 会变化的事实与其时效区间
- 需要关系推理的知识

典型例子：

- “用户现在偏好 light mode”
- “联系人 A 最近一次互动时间”
- “某账号在 1 月和 3 月对同一议题的立场变化”

不要把 Memory 简化成：

- 单纯的持久化
- 只要有向量库就算 memory
- 不带过期和时序处理的事实堆积

## 六、和案例的对应关系

### `digital-brain-skill`

- Tool：4 个 automation scripts
- File：模块目录、JSONL、模板、运营文件
- Memory：联系人、内容历史、目标、运营指标等长期状态

### `x-to-book-system`

- Tool：`x_data_tool`、`memory_tool`、`writing_tool`
- File：raw data、analysis、drafts 的阶段化落盘
- Memory：temporal knowledge graph 维护关系和时间有效性

## 七、最常见的混淆

### 混淆 1：把 tool output 一直留在 context 里

正确做法：

- 让 Tool 产生结果
- 把大结果落到 File
- 必要时把高价值部分写入 Memory

### 混淆 2：把文件存储直接当成 memory 设计完成

正确做法：

- 先确认有没有跨会话检索问题
- 再决定是否需要实体层、时序层、关系层

### 混淆 3：为了“方便”把记忆逻辑写进工具描述

正确做法：

- Tool 定义行动入口
- File 提供外置承载
- Memory 负责长期知识语义

## 八、实战判断句

遇到一个信息对象时，先问三句：

1. 这是要让我现在去做一件事，还是只是要保留结果？
如果是动作入口，优先归 Tool。

2. 这是大体量、低频回看、需要按需读取的内容吗？
如果是，优先归 File。

3. 这是未来还要跨时间、跨任务、跨实体稳定取回的知识吗？
如果是，优先归 Memory。
