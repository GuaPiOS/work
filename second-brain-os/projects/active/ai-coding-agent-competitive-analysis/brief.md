# Brief

## 背景

老板希望依托当前的产品方案，改造出一个 `TMR 全无人套件`。这意味着本项目不只是做市场盘点，还要为“现有产品改造是否可行、应该往哪条产品路径走、优先参考谁”提供研究基础。因为当前背景和方案细节还未完全沉淀，第一步必须先把研究问题、服务对象、时间范围和输出物固定下来，避免后续资料收集失焦。

## 研究目标

建立一套可复用的 `TMR 全无人套件` 竞品分析框架，明确过去 1 年重点竞品有哪些、它们覆盖哪些使用场景、各自采取什么技术路径和商业化方案，以及基于当前产品方案改造的可行性应如何判断，为后续的可行性报告、竞品矩阵、长报告和选型建议提供 canonical 起点。

## 服务对象

你自己、产品团队和老板。第一版重点服务“如何定义研究对象、如何搭比较框架、怎样评估当前产品改造路线”，而不是立即给出最终定案。

## 时间范围

- 外部资料优先时间窗口：最近 `1` 年，默认以 `2025-03-17` 到 `2026-03-17` 为主
- 历史回溯原则：仅在解释技术路线、产品演进拐点或商业化变化时追溯更早资料
- 研究对象原则：优先纳入当前仍可验证、与无人饲喂 / TMR 改造直接相关的产品和方案

## 输出物

1. 可行性报告
2. 竞品矩阵 artifact
3. 长报告
4. 选型建议
5. `artifacts/current/20260317_framing_tmr-unmanned-suite-competitive-analysis_research-frame-v1.md` 作为当前 canonical 入口
6. 后续进入 `context/research/` 和 `context/tool_outputs/` 的来源包、研究笔记和长输出

## 成功标准

- 研究背景、目标、时间范围、服务对象和输出物已经固定，后续资料收集时不再重新定义问题
- 第一版竞品名单和研究维度足以支撑后续的来源采集、竞品矩阵和可行性判断
- 已有首个 canonical artifact，可作为下一轮会话和后续研究的默认入口

## 约束

- 这个线程只服务当前竞品分析项目
- 不修改其他项目
- 不修改 `brain/` 和 `state/`，除非用户明确要求
- 外部原始资料和长输出优先进入 `context/tool_outputs/` 或 `context/research/`
- 阶段性结论优先写入 `artifacts/current/`，不散落在聊天里
